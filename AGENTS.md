# AGENTS.md - AI 接手指南

> 本文件专为 AI 助手设计，帮助快速理解项目结构并正确进行开发协作。

## 项目定位

**墨痕教育自动化录入系统** - 为 WPS「墨痕题库」插件定制的自动化工具链。

核心场景：教辅行业老师将试卷的「题目」和「答案+解析」批量录入 WPS 墨痕题库插件。

## 开发规范

### 必须遵守

1. **修改前必须向用户确认**，得到同意后才动代码
2. **不要动 `wps_helper.py`** - 底层 WPS 连接模块，保持稳定
3. **修改后建议 git commit** - message 写清改了什么、为什么改
4. **新模板开发完毕后必须在 `main.py` 的 `TEMPLATES` 列表中注册** - 仅更新 `__init__.py` 是不够的，`main.py` 不依赖 `__all__`，需手动添加
5. **阅读待清洗文档时，主动识别解析续行模式** - 收到用户的 docx 样本后，必须检查解析块中是否存在 `①②③④`、`A：B：C：D：`、`故本题选X。` 等续行格式，并主动验证清洗逻辑能正确收集这些续行，而不是只收集 `【解析】` 首行
6. **[强制] 修改代码后必须先测试验证，再向用户报告** - 这是最高优先级规则，违反将导致用户信任崩塌
   - 修改代码后，**禁止**立即向用户说"修改完成"或"已修复"
   - **必须**先编写测试脚本或使用实际文档运行测试
   - **必须**确认控制台输出符合预期（无报错、逻辑正确）
   - **必须**在测试全部通过后，才能向用户报告结果
   - 如果无法运行完整测试（如依赖 WPS COM），至少要编写离线单元测试验证核心逻辑
   - **必须**在测试结束后清理测试产生的临时文档（如 `_temp`、`test_output`、临时清洗结果、临时审核清单等），避免工作区再次被测试产物污染
   - **记住**：用户宁愿等你多花 2 分钟测试，也不愿看到未经验证的"已完成"报告
7. **[强制] 每次问题修复或规则调整后必须归档**
   - 记录到项目根目录 `问题归档/`
   - 先更新模块 `INDEX.md`，再补单问题记录
   - 采用“渐进式披露”结构：总索引 → 模块索引 → 单问题记录 → 深层样例/命令
   - 归档内容至少包含：现象、根因、改动点、测试样例、测试命令、结论、下次排查入口
8. **[强制] 记忆与归档采用双轨制**
   - 项目内 `问题归档/` 只保留代码问题的改动证据
   - 外部长期记忆库固定为 `../墨痕教育-记忆库/`
   - 用户说 `检查归档` 时，必须先判断是否需要归档，再更新长期记忆库；如涉及代码问题，再同步检查项目内 `问题归档/`
   - 长期记忆、skill 与归档辅助文件一律放在 `E:`，不要新增 `C:` 盘存储
9. **[强制] 开发前先复盘历史经验**
   - 开发新任务、修 bug、做模板、接新学科/新学校项目前，优先使用 `E:\CODEX.projection\.codex\skills\reviewing-project-history\SKILL.md`
   - 先看 `AGENTS.md`、`问题归档/INDEX.md`、`格式处理/模板开发经验记录/README.md`、`../墨痕教育-记忆库/INDEX.md`
   - 历史经验只作风险提示与验证线索，和当前代码冲突时以当前代码与真实样本验证为准
10. **[强制] 原题文档属于不可变输入**
   - 禁止覆盖、保存或写回原题，禁止改变原题的排版、字号、页边距、表格、图片、标题和段落顺序
   - 原题只允许被只读扫描、建立结构画像、生成外部题块/动作计划，以及在 WPS 中执行选区和按键
   - 题目画像、结构指纹、审核报告和动作计划必须放在原题之外；只有答案文档允许生成 `_已清洗.docx` 等派生产物
   - 架构分析基线见 `docs/墨痕教育架构问题工程思维分析拆解.md`
11. **[强制] 预检产物以 JSON 为机器真值**
   - `DocumentProfile.json` 和 `ActionPlan.json` 是机器事实与执行契约；Markdown 只能由 JSON 派生，用于人工审核，不得反向作为执行输入
   - 第一阶段 PoC 固定 `execution_enabled=false`；在 WPS Range 绑定、审核门禁和受控执行器完成前，不得把预演计划直接改成真实 F1/F2/F3/F4 执行
   - `DocumentProfile` 1.1 的题内角色当前只作证据；`automatic_exclusion_enabled=false`，不得让角色结果绕过文档族阈值和审核门禁直接删除段落
   - `DocumentFamilyReport` 1.0 当前只作 P1a 批次建议；`classification_mode=advisory_only`、`automatic_rule_binding_enabled=false`、`production_execution_enabled=false`，不得把候选文档族直接绑定到生产规则或 WPS 执行
   - P1b 的 macOS Quick Look 结果固定 `page_truth_authority=false`，只供开发预览；只有 Windows WPS COM，或与 `--pdf-input` 同时显式传入 `--attest-wps-export` 的 WPS PDF，才能作为生产页面真值；52/52 人工审核完成前不得进入 P1c

### 代码风格

- 正则规则放入对应配置文件，不散落在逻辑代码中
- 保持各模块入口脚本的独立性
- 测试文件（`test_*.py`）是调试工具，可直接运行验证

2. **必须先打开 WPS** 并加载「墨痕题库工具」侧边栏
3. 脚本通过 COM 接口连接 WPS，按 F1/F2/F3/F4 触发插件动作


## 用户反馈优先级

当用户报 bug 时，按以下优先级排查：

1. `答案录入/answer_input.py` （当前录入行为问题）
2. `格式处理/main.py` + `格式处理/格式模板库/` （格式清洗与审核问题）

## 注意事项

- F1/F2/F3/F4 生产录入仅支持 Windows；macOS 支持开发、测试和 Quick Look 只读视觉预览
- `wps_helper.py` 在两个目录各有独立副本（历史遗留），保持各自独立
- 答案录入模块通过 WPS COM 读取活动文档段落并建立实际选区，同时使用 `python-docx` 做离线解析、审核和临时连续文档处理
- 无解析的题目，格式清洗阶段会在解析行末尾补空格，确保 F3 正常跳转

## 跨项目复用规则

以下规则来自 2026-04 地理项目的连续实战，后续接手新学科或新学校项目时，默认先按这套规则执行，不要从零试错。

### 架构优先级

- 架构进度以 `docs/墨痕教育架构问题工程思维分析拆解.md` 为准；截至 2026-07-18，6 个根问题为 1 个已解决、5 个部分解决、0 个未解决（严格完成 1/6，已进入工程解决 6/6）
- P1a 已能从 Profile 1.1 生成候选文档族、代表样本和异常候选；P1b 跨平台页面视觉审核与校准门禁框架已落地，但 52/52 Windows WPS 真实批次尚未审核，P1c 文档族规则快照也未形成，因此不得视为生产分族完成
- 学科逻辑优先采用“三大核心 + 学科覆盖层”，不要直接把系统拆成一套套完全独立的地理/生物/语文引擎
- 三大核心仍是 `英语 / 理科 / 文科`，具体学科差异优先放到 `subject_overlay`
- 当前已验证可行的模式是 `文科 + geography`；后续若扩生物、历史、政治，也优先复用 overlay 机制
- 学科特例不要散落在 `core_parser.py` 的临时 `if/else` 里，应尽量集中到覆盖层或共享内核

### 题目录入规则

- 材料题如果出现明确范围，如 `完成3-5小题`、`据图完成6-8小题`，必须按题号范围截断，只吞并范围内小题，遇到下一个超范围题号立刻停止
- 主观大题如 `13. ……回答下列问题。(1)…(5)…`，`F1` 默认按一道大题录入，不按小问拆成多道题
- 地理图表题、地图题、示意图题如果题干和选项之间夹有图片或表格，优先保证整题边界正确，再处理展示样式
- 所有学科、学校和项目的表格题统一保留原生表格：直接整块框选题干、表格和选项后按 F1 录入，不再转纯文本或创建临时文档
- 表格内的小数、百分数、数据单元默认不能当题号起点；如果日志里“发现题号数量”和“准备录入题数”明显不一致，优先排查表格元数据是否丢了
- 大题内部的 `【独特的自然环境】`、`【一粒种子的远征】` 这类板块标题，默认视为题内结构，不视为截断边界
- 低置信度题默认“后台标记、不中断录入”，不要每题停下来等用户回车确认
- 风险提示必须说中文，不要把 `image_related_question`、`sparse_options` 这类内部码直接暴露给用户
- 如果某类装饰题号在一个项目里反复出现且不是单卷个例，例如 `★11.`、WPS 文本层里的 `/11.`，应升级为共享前缀兼容，不要继续靠白名单补丁
- 语文资料若出现“非编号阅读提示 → 材料正文 → 第一题题号”，且材料只应随首题录入，优先在项目级 `subject_overlay` 配置 `leading_context_patterns`，由共享内核生成 `LEADING_CONTEXT` 题块；提示行本身不录入，后续顶层题仍独立，WPS 最终选区不得再次截断已确认的前置材料
- 众美对点练这类“一篇文章 + 多道所属题”的完整阅读题，项目覆盖层必须开启 `group_leading_context_questions`，将文章与下一个阅读提示或顶层大板块边界前的全部所属题合并为一个 `LEADING_CONTEXT` 题块；“（一）高考专练 / （二）典题专练”等题内子板块既不得截断题块，也不得进入最终F1选区，无原生表格时执行层应在不保存的临时富文本副本中剔除标题后整组录入，普通媒体和VML装饰对象随富文本保留，如标题与原生表格共存则停止并复核；题答核对应使用 `infer_grouped_question_ids()` 展开内部题号，不能只比较题块数

### 学科特例处理

- `G1.`、`C2.`、`Y6.`、`【G】1.` 这种前缀题号，不要做成全局通用规则
- 是否保留这类前缀，默认放到具体学科覆盖层或白名单文档里处理
- 如果是某一批卷子的个例，就按该批文档白名单保留；不要为了一个案例污染全部题目录入规则
- 文件名不含“地理”时，也要结合正文关键词、题型特征做内容嗅探，避免地理卷误判到理科

### 答案清洗与录入规则

- 原始答案同时存在“表格答案”和“正文答案+解析”时，默认正文优先，表格只补正文缺失题号，不重复摘录
- 清洗结果里，试卷标题、题型标题、红色说明性抬头默认不保留
- 连写答案、表格答案、材料题小问答案都要在清洗阶段标准化，不要把识别压力留到 `answer_input.py`
- `answer_input.py` 直接录入前，标准化答案文档必须有审核状态文件；用户手改过 docx 后，要视为审核状态失效，需要重新过审核门
- 缺少解析的答案块会被系统阻断；如果业务允许空解析，也要显式补 `解析：` 占位，而不是让块结构悬空
- 不是所有带 `(1)(2)(3)` 的答案都应该拆成 `F4`；如果插件题型实际只留一个答案框，应改为整段 `F2` 录入，并把该策略做成项目/文档级规则
- `①②③…` 统一视为一道答案内部的列点，不作为 `F4` 小问标志；只有明确的 `(1)(2)…` / `（1）（2）…` 小问或完整阅读题分组才进入 `F4` 流程
- 数学或理科答案里，**能稳定转成纯文本的内容可以留在答案区；公式对象、富文本风险或会乱码的内容转入解析区承载**，不要强塞进答案框
- 数学斜体 Unicode 字母（如 `𝑥 / 𝑦 / 𝑏 / 𝑘`）先在清洗阶段归一化成普通 `x / y / b / k`，再判断是否需要转解析；不要把这类字符误判成公式对象
- 数学富文本模板清洗后如果需要保公式，默认跳过纯文本题答重写链路；一旦误走重写链路，优先怀疑公式对象被写没，而不是先怀疑模板没命中
- 众美高三语文对点练答案使用 `tools/clean_zhongmei_chinese_answers.py` 专项脚本；运行时先做整批只读预检，只处理严格命名的 `对点练案N.docx`，遇到未知答案标记、表格答案或题号不连续必须整批停止，无解析题显式补空白 `解析：`
- 众美高三语文文言文答案使用 `tools/clean_zhongmei_classical_chinese_answers.py` 专项脚本；运行前严格递归配对整批题答，双篇保留各篇原题号重置，仅 `20《苏武传》` 将原 `1、2、5、6` 修正为 `1、2、3、4`；翻译语句统一按“全部 `(n)答案` 连续排列 → 单一 `解析：` → 全部 `(n)得分点`”输出，录入时每个小题使用 F4、整组解析只使用一次 F3，即使只有一个翻译小题也保留 `(1)` 并走 F4；得分点移入解析，已知图片字按哈希恢复，遇未知图片或结构异常整批停止
- 众美项目的路径上下文识别只依赖稳定前缀 `众美-高三-语文`，不得绑定人员后缀；项目总目录改名后必须同步默认路径并重新刷新与绝对路径签名绑定的审核状态，不能直接沿用旧状态文件
- 众美文言文答案录入若发现 WPS COM 的 `ActiveDocument` 仍位于项目 `文言文/` 原题目录，应映射到 `答案/文言文答案/` 下同层级、同名的 `_已清洗.docx`；必须先验证目标文件存在且审核门禁通过，才能复用或打开并激活目标文档，禁止为绕过门禁而直接扫描原题
- 众美已清洗答案中的训练标题、题型小标题和“阅读下面……完成题目”提示只供人工阅读；答案解析器必须把这些标题视为题块边界，F2/F3/F4 选区不得包含标题，即使旧合并文档把标题粘在解析末段也要剥离

### 输出与工作区管理

- 开发辅助产物，如 `_审核清单.md`、调试摘要、归档补充文档，默认放项目目录或 `E:` 盘长期目录，不要放进用户的题目源目录
- P1b PDF、页面 PNG、页面/视觉/校准 JSON 及 `BatchSourceManifest.json` 必须放在原题和 Git 仓库之外，不得提交业务路径或真实文档派生事实
- 清洗后的答案文件，每份只保留一个最终版 `*_已清洗.docx`，中间版本及时清理
- 面向用户实际录入的目录尽量只保留“原题 / 原答案 / 最终清洗版”，不要混入开发过程文件

### 开发与验证习惯

- 每次修规则，至少做两层验证：离线测试覆盖核心逻辑，真实样本文档做定向复核
- 优先补失败用例，再修代码，避免只靠人工口头确认
- 测试结束后清理临时 docx、临时审核清单、临时输出目录，避免污染下一轮判断
- 同一类问题修到第三次时，应优先考虑抽象成共享规则或覆盖层，不要继续补局部特判
