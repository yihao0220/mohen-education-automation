# AGENTS沉淀地理项目跨项目复用规则

## 现象

- 2026-04-20 到 2026-04-22 的地理项目里，规则调整非常密集，涉及题号前缀、材料题范围、表格题、内部板块标题、答案审核门禁、开发产物落盘位置等多个层面。
- 这些有效结论原本散落在聊天记录和单问题归档里，下次换一个学校或新学科项目时，新的接手 Agent 很难快速拿到“哪些是通用经验、哪些只是地理个例”。

## 根因

- 现有 `AGENTS.md` 侧重项目结构和通用开发规范，缺少“2026-04-20 到 2026-04-22 这轮真实项目已经验证过的复用规则”。
- 地理项目中多次出现“先修一个局部 bug，随后又发现它应上升为通用工作流或覆盖层规则”的情况，如果不二次沉淀，后续开发仍会重复绕路。

## 改动点

1. 在项目根目录 [AGENTS.md](../../AGENTS.md) 新增 `跨项目复用规则` 章节。
2. 将本轮地理项目中最稳定、最值得复用的结论收敛为六类规则：
   - 架构优先级
   - 题目录入规则
   - 学科特例处理
   - 答案清洗与录入规则
   - 输出与工作区管理
   - 开发与验证习惯
3. 明确区分：
   - 应提升为共享规则的内容
   - 只应放在 overlay 或白名单里的学科个例
   - 仅作为开发流程约束存在的协作规则

## 本次沉淀的核心结论

- 学科扩展优先走“三大核心 + overlay”，不要直接裂变成多套主引擎。
- 材料题范围必须有“题号范围约束”，不能只靠读到下一题前一直吞并。
- 大题内部板块标题通常是题内结构，不是边界。
- 表格题优先按 `.docx` 结构重建，不要过度依赖 WPS 视图顺序。
- `G/C/Y` 前缀题号属于学科或文档级特例，不应污染全局规则。
- 清洗后的答案直录前，审核状态文件和答案块结构完整性缺一不可。
- 开发辅助产物和中间版本不能继续堆在用户的题目源目录里。

## 测试与核对

- 核对 [AGENTS.md](../../AGENTS.md) 已新增 `跨项目复用规则` 章节，内容可直接给后续 Agent 继承。
- 核对 [题目录入索引](./INDEX.md) 与 [总索引](../INDEX.md) 已加入本条记录。

## 结论

- 这次不是新增代码能力，而是把已经验证过的一批高频规则正式升级为项目级接手约束。
- 后续再做新的学校项目或新学科项目时，应先看 `AGENTS.md` 的这段沉淀，再决定哪些规则直接沿用、哪些需要局部覆写。

## 下次排查入口

- 如果是题目录入边界问题，先看 [AGENTS.md](../../AGENTS.md) 的 `跨项目复用规则`，再按需进入 [题目录入索引](./INDEX.md) 查看具体案例。
- 如果是答案清洗或审核门禁问题，再联动查看 [格式转换索引](../格式转换/INDEX.md) 和 [答案录入索引](../答案录入/INDEX.md)。
