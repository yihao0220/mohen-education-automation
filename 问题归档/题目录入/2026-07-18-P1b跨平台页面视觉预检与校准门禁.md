# P1b跨平台页面视觉预检与校准门禁

## L1 摘要
- 模块：题目录入 / 只读预检旁路
- 日期：2026-07-18
- 状态：开发框架已完成并复核；Windows 52 份生产校准待执行
- 结论：Mac 可无鼠标生成 DOCX 连续视觉预览与区域 JSON；Windows WPS 只读 PDF 是唯一生产页面真值。两类结果由硬门禁区分，均不会触发 F1/F2/F3/F4。

## L2 现象与根因
- 现象：仅靠 `python-docx`/OOXML 能发现图片节点，却不能可靠区分章节横幅、`对点训练`、`综合强化` 等装饰图与题目图；用屏幕滚动截图又受窗口、缩放和鼠标状态影响。
- 根因：P1a 缺少稳定的页面视觉事实、人工角色真值、题图绑定和整批校准门禁；Mac 与 Windows 的渲染职责也没有分开。

- 核心规则：Mac Quick Look 固定 `renderer_role=development_preview`、`page_truth_authority=false`；Windows WPS COM 固定只读打开、关闭不保存并复核源哈希，才可作为 `production_truth`。
- 栅格化：默认使用 `pypdfium2`，保留显式 `pdftoppm` 兼容入口；每次渲染先删除目标目录旧 `page-*.png`，防止陈旧页面混入。
- 生产真值加固：WPS COM 不再回退到 `Word.Application`；任意 `--pdf-input` 默认只作开发预览，只有同时传入 `--attest-wps-export` 才记录为用户确认的 WPS PDF。
- 跨机开发：Windows 可用 `setup_windows.ps1 -Development` 安装开发依赖，并用 `verify_windows.ps1 -RunTests` 只读检查 WPS COM 注册和离线测试；自检不启动 WPS、不打开文档、不执行按键。
- 持续验证：`.github/workflows/p1b-windows.yml` 在全新 Windows runner 上执行开发安装和 P1b 离线回归；工作流权限固定只读，第三方 Action 固定到已核验提交 SHA，不把无 WPS 的 runner 当成生产页面真值。
- Windows 编码修复：首次 runner 暴露 Windows PowerShell 5.1 会把无 BOM 的 UTF-8 中文脚本误读成乱码并报字符串未闭合；两个 `.ps1` 已统一增加 UTF-8 BOM，`test_windows_scripts.py` 固化编码门禁。
- Windows CLI 输出修复：第二次 runner 暴露 Python 在重定向 stderr 时会按旧代码页把中文写成字面量 `\\uXXXX`；四个 P1b CLI 统一调用 `configure_utf8_stdio()`，测试子进程固定按 UTF-8 解码并继续断言真实中文。
- 2026-07-18 知识收尾：README、AGENTS、P1b 设计/实施计划、架构账本、工作区约定和回归样本说明已按当前实现对齐；真实基线改为明确的本机忽略资产，P1b 文件名统一为 `VisualRoleReview.json`、`VisualReviewSummary.json`、`BatchSourceManifest.json`，并补入 `.gitignore`，避免业务路径和真实文档派生事实误提交。
- 测试样例：两页合成 PDF、合成 DOCX/图片、伪 WPS COM 生命周期、视觉角色阻断、阈值误拆误并、批次只读副本、真实未来高二生物作业1。
