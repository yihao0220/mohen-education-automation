# DOCX只读画像与F1预检动作计划PoC

## L1 摘要

- 模块：题目录入 / 共享内核
- 日期：2026-07-16
- 状态：已完成并复核
- 结论：建立了“原始 DOCX 只读 → 原生结构扫描 → Docling 旁路对照 → `DocumentProfile.json` → `ActionPlan.json` → 派生 Markdown”的最小链路。PoC 固定 `execution_enabled=false`，不连接 WPS，不执行 F1/F2/F3/F4。

## L2 现象与根因

### 现象

- 现有题目录入能直接从 DOCX/WPS 识别题块，但缺少真实执行前可保存、可比较、可人工审核的机器契约。
- 批量规则调整时主要靠日志和测试断言，难以直接观察表格、媒体、公式、原始段落坐标与最终动作之间的关系。
- 原题属于不可变输入，不能为了统一结构而改写、另存或扁平化原生表格。

### 根因

- `DocNode` / `QuestionUnit` 已经存在，但没有文档级 `DocumentProfile` 和动作级 JSON sidecar。
- Docling 能补充结构识别，却不能单独证明其元素可映射回 WPS 原始段落，因此不能替换现有坐标扫描器。
- 机器事实和人工报告此前没有明确分工，容易让 Markdown 或临时日志承担执行输入职责。

## L3 改动点

### `shared_core/document_preflight.py`

- 读取前后计算 SHA256；若原文件发生变化，立即拒绝生成计划。
- 原生扫描统计段落、表格、Drawing、VML、公式、标题和表格锚点。
- Docling 作为可选旁路，只保存稳定的结构摘要与差异，不保存完整转换副本。
- 复用现有 `scan_docx_nodes()`、`QuestionUnit.source_span` 和 `infer_grouped_question_ids()` 编译 F1 预演动作。
- `source_ref` 同时记录虚拟节点、原始段落、原生表格、媒体和公式段落。
- JSON 为权威机器事实；Markdown 只由 JSON 单向生成。

### `tools/build_document_preflight.py`

- 支持一次输入一个或多个 DOCX。
- 所有产物写入显式 `--output-dir`，不写回源目录。
- 支持 `--no-docling`，第三方转换不可用时仍可运行原生画像。

### 测试与依赖

- `test_document_preflight.py` 覆盖只读哈希、表格锚点、JSON/MD、UTF-8 黄金 JSON 和 Docling 结构对照。
- `requirements-poc.txt` 固定 Docling 2.113.0、pytest 9.1.1、pytest-regressions 2.11.0；它不是现有 WPS 主入口的强制依赖。
- 设计入口：`docs/plans/2026-07-16-document-preflight-poc-design.md`。
