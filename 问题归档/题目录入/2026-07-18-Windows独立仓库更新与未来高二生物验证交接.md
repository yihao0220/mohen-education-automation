# Windows独立仓库更新与未来高二生物验证交接

## 现象

- 用户原有代码目录是 `E:\PYTHON\practice\墨痕教育`，但 `git rev-parse --show-toplevel` 实际返回 `E:/PYTHON/practice`。
- 在子目录内执行 Git 命令时，状态会同时出现 `../课表网站/` 等无关项目；这意味着覆盖范围不是“墨痕教育”子目录，而是整个上级仓库。
- 上级仓库没有本项目的 `origin`，因此出现 `origin does not appear to be a git repository`；Windows 保留名 `nul` 还导致 `Unlink ... failed` 交互提示。

## 根因

- 更新前只看了 PowerShell 当前目录，没有先验证真实 Git 顶层目录。
- 旧代码、其他项目和业务文件位于同一个过大的父级仓库中，不能把“当前所在子目录”等同于“Git 操作范围”。
- 直接给父级仓库添加本项目远端并执行覆盖，会把无关项目一起纳入风险范围。

## 改动点

1. 先用备份分支 `backup/windows-before-biology-20260718-203905` 将父级仓库跟踪状态恢复到 `fecf39e`，并移除临时添加的错误 `origin`。
2. 保留 `stash@{0}` 和备份分支，不执行 `stash pop`、`git clean` 或删除未跟踪业务文件。
3. 将 GitHub 项目完整克隆到独立目录 `E:\CODEX.projection\墨痕教育`，安装时核对版本为 `18b9b05`。
4. 用户运行 `scripts\setup_windows.ps1 -Development`，确认显示“Windows 开发与生产环境已就绪”。
5. 新增 [Windows更新与未来高二生物验证手册](../../docs/Windows更新与未来高二生物验证手册.md)，同步 README 和 AGENTS 的 Windows 安全门禁。

- 父级仓库回滚后：`HEAD=fecf39e`，状态只剩未跟踪文件，未继续执行破坏性清理。
- 独立仓库：`git rev-parse --short HEAD` 返回 `18b9b05`。
- Windows 安装：用户确认脚本输出“Windows 开发与生产环境已就绪”。
- 交接当时尚未取得 `verify_windows.ps1 -RunTests`、生物专项或真实 WPS F1/F2/F3/F4 的本机结果，因此不能把“安装完成”升级为“生产验证完成”。
