# 审核状态跨机携带与 Windows 打包

## 现象

未来高二生物的 `_已清洗.docx` 和 `_已清洗_审核状态.json` 在 Mac 上为 `approved`，通过 ZIP 发到 Windows 后却立即变成 `stale`，答案录入持续提示“答案文件在自动检查后发生变更”。同时 Mac Finder 生成的 `答案.zip` 把内部中文路径写成乱码，且未设置 UTF-8 文件名标志。

## 根因

- 旧审核签名包含绝对路径、文件大小和 `mtime_ns`。ZIP 解压到另一台电脑后，即使 DOCX 字节完全不变，路径和修改时间也必然变化。
- 修改时间不是内容证据；它既会因复制误报，也可被恢复，不能可靠地发现同大小内容修改。
- Mac 默认压缩包还夹带 `.DS_Store` 和 `__MACOSX`，不适合作为 Windows 生产交付物。

## 改动点

- `shared_core/review_gate.py` 将审核签名升级为 `version=2 + size + SHA256`；绝对路径和修改时间不再参与签名相等判定。
- 签名计算前后会复核文件大小和修改时间，防止在计算 SHA256 时文件正被同时修改。
- 旧版状态缺少内容哈希，不做不安全的自动迁移，必须重新清洗和自动检查。
- 新增 `tools/package_future_biology_answers.py`：打包前检查每份答案门禁，只包含 `答案/已清洗` 和 `答案/审核清单`，路径统一 NFC + UTF-8，排除 Mac 元数据，逐文件复核包内 SHA256。
- 外层包名固定为 `future_biology_cleaned_answers_windows.zip`，避免传输工具对中文附件名二次转码。

## 测试样例

- 同一 DOCX 和状态复制到另一绝对路径，并修改文件时间：仍为 `approved`。
- 修改内容但保持文件大小和原修改时间：必须为 `stale`。
- 旧版路径/mtime 签名：明确阻断，不静默升级。
- 合成 Windows ZIP：所有非 ASCII 路径均有 UTF-8 标志，无 `.DS_Store` / `__MACOSX`，解压后门禁仍通过。
- 真实生物包：42/42 份在临时 Windows 路径模拟解压后仍 `approved`；抽取一份做同大小字节修改后变为 `stale`。

## 测试命令

```bash
.venv/bin/python -m pytest -q -p no:cacheprovider test_review_gate_portability.py test_future_biology_answer_package.py test_future_biology_answer_clean.py
.venv/bin/python tools/package_future_biology_answers.py "/绝对路径/未来-高二-生物" --expected-docx-count 42 --overwrite
```

## 结论

审核门禁现在绑定答案内容，不再绑定某台机器的路径和时间。移动、改名、ZIP 解压不会误报；任意字节修改仍会失效。Windows 必须同时更新到新门禁代码，并用新压缩包覆盖旧业务派生文件。

## 下次排查入口

1. 跨机后显示“旧版文件签名”：状态 JSON 没有 `answer_signature.version=2`，需重新清洗。
2. 跨机后仍显示“文件变更”：先比较 DOCX SHA256，不要手改状态 JSON。
3. 文件名乱码：确认发送的外层包名为英文，并使用专项打包脚本，不用 Mac Finder 直接压缩中文目录。
4. 同一文件频繁检查较慢：先测 SHA256 实际耗时，不降级回 mtime 签名。
