# 标准答案录入改为系统自动检查放行

## 现象
- 现有流程里，答案清洗后虽然会生成 `*_审核清单.md`，但缺少一个“程序自己判断是否可继续录入”的放行结果。
- 用户要求改成自动检查：答案清洗完成后，系统要立刻自检一遍；检查通过就保留清洗结果并允许继续录入，检查不通过才拦下。

## 根因
- 系统只有“审核报告”，没有“自动检查状态”。
- 缺少一个可持久化、可失效的放行凭证，导致程序无法区分“自动检查已通过”和“只是生成过审核清单”。
- 标准答案文件在检查后如果再次被修改，旧检查结果也不会自动失效。

## 改动点
- `shared_core/review_gate.py`
- 新增自动检查状态模块，围绕每份 `*_已清洗.docx` 生成 `*_审核状态.json`
- 支持四种核心状态：`pending`、`approved`、`rejected`、`stale`
- 状态文件记录答案文件签名（`size` + `mtime_ns`），一旦答案文件被改动，旧放行结果自动失效
- `main.py`
- 清洗+审核完成后先初始化状态，再根据审核报告自动写入 `approved` 或 `rejected`
- 全流程录入改为“录题目 + 清洗 + 自动检查 + 通过则继续录入答案”
- 菜单 `5` 保留为手动覆盖入口，只在需要人工兜底时使用，不再是主流程必经步骤
- `答案录入/answer_input.py`
- 单独运行答案录入工具时，也会检查当前文档是否存在有效的自动检查放行状态
- `test_zero_error_pipeline.py`
- 新增自动检查回归：无高风险时自动 `approved`，有高风险时自动 `rejected`
- 保留状态失效回归：`approved -> 文件修改后 stale`

## 测试样例
- 新建临时 `答案_已清洗.docx`
- 初始化状态后，应为 `pending`
- 自动检查通过后，应写成 `approved` 并允许录入
- 自动检查失败后，应写成 `rejected` 并拒绝录入
- 通过后再修改答案文件，状态应自动失效并变成 `stale`


## 结论
- 现在不是“生成了审核清单就算过了”，而是系统会在清洗后立刻自动检查，并把结果写进有效的 `*_审核状态.json`。
- 自动检查通过时，状态为 `approved`，清洗后的答案直接保留并允许后续录入。
- 自动检查不通过时，状态为 `rejected`，保留清洗结果和审核清单，但阻止继续录入。
- 如果答案文件后来又被改动，旧放行结果自动失效并变为 `stale`，必须重新跑清洗和自动检查。

## 2026-07-18 后续修订

本记录中的 `size + mtime_ns` 签名已被 [审核状态跨机携带与 Windows 打包](./2026-07-18-审核状态跨机携带与Windows打包.md) 取代。现行签名为 `size + SHA256`：移动或解压不失效，内容变化仍失效。
